03Piece 1 — nginx upstream health check

Most important piece. Stops nginx from routing to a dead FastAPI process, and exposes a public /healthz for external monitors.

Add to your nginx site config

Typically /etc/nginx/sites-available/mygenie (or equivalent).

# Define the FastAPI upstream with fail-fast semantics.
# After 3 consecutive failed requests within 30s, nginx marks it "down"
# for 30s and stops sending traffic. Recovery is automatic.
upstream fastapi_backend {
    server 127.0.0.1:8001 max_fails=3 fail_timeout=30s;
    keepalive 32;
}

# Health-check location — proxies to FastAPI /api/healthz with tight timeout.
# Cloudflare / uptime services should point at this path.
location = /healthz {
    proxy_pass http://127.0.0.1:8001/api/healthz;
    proxy_read_timeout 3s;
    proxy_connect_timeout 2s;
    proxy_send_timeout 2s;
    access_log off;
    add_header Cache-Control "no-cache, no-store" always;

    # Rate limit — 60 req/min/IP. Requires the limit_req_zone below.
    limit_req zone=healthz burst=10 nodelay;
}

# Main API routing — nginx auto-fails-over between upstreams if you have multiple.
location /api/ {
    proxy_pass http://fastapi_backend;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    # Aligned with CR-004 client-side caps (FE waits 15s, BE has 20s room).
    proxy_read_timeout 20s;
    proxy_connect_timeout 5s;
    proxy_send_timeout 20s;

    # Fast-fail cascade on upstream error.
    proxy_next_upstream error timeout http_502 http_503;
    proxy_next_upstream_tries 2;
    proxy_next_upstream_timeout 5s;
}

Add once in the http { } block

Typically /etc/nginx/nginx.conf.

# Rate-limit zone for /healthz — 1MB memory, 60 requests per minute per IP.
limit_req_zone $binary_remote_addr zone=healthz:1m rate=60r/m;

Apply

sudo nginx -t                    # syntax check
sudo systemctl reload nginx      # zero-downtime reload

Verify

# Public health endpoint responds
curl -s https://YOUR_DOMAIN/healthz
# → {"ok":true,"mongo":"up"}

# Rate limit works (11th request in a second should 503)
for i in {1..15}; do curl -so /dev/null -w "%{http_code} " https://YOUR_DOMAIN/healthz; done
# → 200 200 200 200 200 200 200 200 200 200 503 503 ...

What this buys you

  • When Mongo dies: FastAPI's /api/healthz returns 503 → nginx sees consecutive failures on /api/* → marks upstream down → returns 502 to clients (fast fail) → auto-recovers when Mongo returns.
  • When FastAPI crashes: Same behavior — nginx bypasses it.
  • Public probe endpoint at /healthz: external uptime services can hit it with 3s timeout.

04Piece 2 — PM2 auto-restart on health failure

Only if you're using PM2 (not just Supervisor). If Supervisor, skip to Piece 3.

Option A — Watchdog script + cron

Create /app/ops/watchdog.sh:

#!/usr/bin/env bash
# Watchdog: restarts FastAPI via PM2 if /healthz is unhealthy for 3
# consecutive minutes. Prevents flapping on brief Mongo blips.
#
# Add to cron: */1 * * * * /app/ops/watchdog.sh >> /var/log/watchdog.log 2>&1

set -euo pipefail

STATE_FILE=/var/run/mygenie-watchdog.count
MAX_FAILURES=3
HEALTH_URL="http://127.0.0.1:8001/api/healthz"
PM2_APP_NAME="mygenie-backend"

count=$(cat "$STATE_FILE" 2>/dev/null || echo 0)

if curl -sf --max-time 3 "$HEALTH_URL" | grep -q '"ok":true'; then
    echo 0 > "$STATE_FILE"
    exit 0
fi

count=$((count + 1))
echo "$count" > "$STATE_FILE"
echo "$(date -Iseconds) unhealthy count=$count"

if [ "$count" -ge "$MAX_FAILURES" ]; then
    echo "$(date -Iseconds) restart triggered (count=$count)"
    pm2 restart "$PM2_APP_NAME"
    echo 0 > "$STATE_FILE"
fi

Make executable + add to root's crontab:

sudo chmod +x /app/ops/watchdog.sh
sudo crontab -e
# Add this line:
# */1 * * * * /app/ops/watchdog.sh >> /var/log/watchdog.log 2>&1

Option B — PM2 ecosystem file

Create/update /app/ecosystem.config.js:

module.exports = {
  apps: [{
    name: 'mygenie-backend',
    script: 'uvicorn',
    args: 'server:app --host 0.0.0.0 --port 8001',
    cwd: '/app/backend',
    interpreter: 'python',

    // Restart if memory usage indicates a leak
    max_memory_restart: '1G',

    // Prevent restart storm
    max_restarts: 10,
    min_uptime: '60s',

    // Log rotation
    out_file: '/var/log/pm2/mygenie-out.log',
    error_file: '/var/log/pm2/mygenie-err.log',
    time: true,

    env: {
      // Real secrets from a secret manager, not this file
      NODE_ENV: 'production'
    }
  }]
};
pm2 start /app/ecosystem.config.js
pm2 save                          # persist across reboots
pm2 startup                       # generate systemd unit for boot

Do not do this

Don't set the watchdog to restart on every health-check failure. Nginx already handles routing-layer fail-over. Thrashing PM2 restarts during a Mongo outage causes worse UX than clean 502s.

05Piece 3 — Supervisor + cron wrapper

Use if you're on Supervisor (like the dev container). Supervisor doesn't natively call HTTP health checks.

Verify supervisor config

/etc/supervisor/conf.d/mygenie-backend.conf:

[program:backend]
command=/root/.venv/bin/uvicorn server:app --host 0.0.0.0 --port 8001
directory=/app/backend
autostart=true
autorestart=true
startretries=5
startsecs=15
stdout_logfile=/var/log/supervisor/backend.out.log
stderr_logfile=/var/log/supervisor/backend.err.log
environment=PATH="/root/.venv/bin:%(ENV_PATH)s"

Key settings:

  • autorestart=true — restarts on unclean exit
  • startretries=5 + startsecs=15 — gives up if it can't stay up 15 s five times (prevents thrash)

Cron watchdog (same as Piece 2 Option A)

Use the identical script from Piece 2, but replace the PM2 restart line with:

sudo supervisorctl restart backend

06Piece 4 — External uptime service

Even with pieces 1-3 in place, you want a third party watching from outside your network. Covers the failure mode where your entire host disappears.

Recommended free-tier options

ServiceFree tierBest for
UptimeRobot50 monitors, 5-min intervalSimple ping + Slack alert
BetterUptime10 monitors, 3-min intervalBetter UI, incident timeline
Cronitor5 monitors, 1-min intervalIf you also monitor cron jobs
PingdomPaid onlyEnterprise SLA reports
Cloudflare Health ChecksWith CF Pro planIntegrates with existing CF edge

Configuration (any of them, same shape)

  • URL: https://YOUR_DOMAIN/healthz
  • Method: GET
  • Interval: 60 s (some free tiers cap at 5 min — still fine)
  • Timeout: 5 s
  • Expected status: 200
  • Expected body contains: "ok":true
  • Alert on: 2 consecutive failures
  • Alert channels: Slack webhook + email + SMS (optional)

07What NOT to do

Don'tWhy
Set proxy_read_timeout 3s on /api/* (not just /healthz)Legitimate requests take longer than 3s (uploads, complex queries). Only the health check should be that tight.
Restart FastAPI on every /healthz 503Thrashes the process during a Mongo outage. Nginx already handles fail-over. The watchdog needs a 3-min sustained threshold.
Set PM2 max_restarts: 100PM2's default (10 restarts / 60s) is protection. If your app crashes more, restarting harder won't help.
Log /healthz to access logUptime monitors poll every 60s = 1440 noise lines/day. Use access_log off;.
Put real Mongo credentials in ecosystem.config.jsUse a secret manager or systemd Environment file with 0600 permissions.
Point external uptime at /api/healthz (with prefix)Better to expose /healthz via nginx — cleaner, and if /api/* routing has a bug you still see the health signal.
Skip the rate limit on /healthzIt's public. Someone will script-hammer it. 60r/m is plenty.

  • Piece 4 — external uptime service · sign up, add URL, test alert to Slack. 10 min
  • Piece 2 or 3 — watchdog cron · deploy script, verify cron runs, test by stopping FastAPI temporarily. 10 min

09End-to-end validation test

Once all pieces are in place, simulate a Mongo outage:

# Block outbound traffic to Mongo
sudo iptables -A OUTPUT -d 52.66.232.149 -p tcp --dport 27017 -j DROP

Watch what happens over ~2 minutes:

  1. Backend logs show ServerSelectionTimeoutError (existing behaviour)
  2. /api/healthz returns 503
  3. Nginx starts returning 502 on /api/* after 3 failures (fast-fail)
  4. Watchdog cron increments failure count
  5. External uptime service alerts within 60s
  6. Watchdog restarts backend after 3 minutes if configured

Recover:

sudo iptables -D OUTPUT -d 52.66.232.149 -p tcp --dport 27017 -j DROP

Within 30s:

  1. /api/healthz → 200
  2. Nginx auto-recovers upstream (fail_timeout=30s)
  3. Traffic resumes normally
  4. External uptime posts "resolved"

If any step doesn't fire

That pipeline has a gap. Investigate before considering the wiring complete.

10Quick-reference runbook commands

Save these somewhere ops can find in an incident (e.g. /app/ops/README.md):

# Health check
curl -s https://YOUR_DOMAIN/healthz
# → {"ok":true,"mongo":"up"}  on green

# Recent Mongo errors in backend logs
sudo tail -f /var/log/supervisor/backend.err.log | grep -E "ServerSelection|NetworkTimeout"

# Restart backend gracefully
sudo supervisorctl restart backend    # supervisor
pm2 restart mygenie-backend            # or PM2

# Watchdog log
sudo tail -f /var/log/watchdog.log

# Force nginx to re-check upstream
sudo nginx -s reload

One-sentence summary

Wire nginx to check /api/healthz, add a cron watchdog with sustained-failure threshold, sign up for an external uptime service — and Mongo outages become auto-detected, auto-routed, and auto-alerted instead of discovered via customer complaints.
