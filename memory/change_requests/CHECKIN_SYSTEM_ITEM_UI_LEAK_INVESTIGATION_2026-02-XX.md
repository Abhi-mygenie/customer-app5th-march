## 6. Where the filter is missing

A single point: **`transformOrderDetails(api)`** at `orderTransformer.ts:167-169`:

```ts
const items = (api.details || []).map(transformOrderItem);
const previousItems = (api.details || []).map(transformPreviousOrderItem);
```

If we filter `api.details` here (or filter the mapped arrays), every downstream consumer (OrderSuccess "Items Ordered", ReviewOrder "Previously Ordered", CartBar count, tax loop) is corrected at once, and `billSummary.itemTotal` is automatically corrected because it uses `previousItems.reduce(...)` further down.

## 7. Filter condition — strongest available

The schema does not expose a categorical flag. The available identifiers ordered by reliability:

| Option | Field | Reliability | Risk |
|---|---|---|---|
| A | `food_details.name` (case-insensitive, trim, normalized) match against `"check in"` / `"check-in"` / `"checkin"` | High enough — owner-confirmed canonical label | Low — implausible a real food item is named exactly "Check In" |
| B | `food_id` (numeric POS id) | Highest — globally stable | **NOT KNOWN at frontend** without backend confirmation; varies per restaurant |
| C | `food_details.category_id` (POS system-item category) | High if it exists | **UNKNOWN value**; backend confirmation required |
| D | Combined: A + (price <= 1) | Slightly safer (system items typically have nominal price) | False negative if POS sets price 0 vs 1 vs other |
| E | Combined: A + zero-tax + zero-addons + zero-variations | Defensively safer, narrows match | Slightly more complex; effectively still A |

**Recommendation:** **Option A**, normalized:
```ts
const norm = (s: string) => (s || '').toString().trim().toLowerCase().replace(/[\s\-_]+/g, ' ');
const isCheckInItem = (foodName?: string) =>
  norm(foodName) === 'check in';
```
Single, narrow string match. Equivalent to how the POS distinguishes the item itself.

> **Risk acknowledgement (per STOP CONDITION):** the only available filter is the item name. A real menu item literally named "Check In" would be hidden. In a food/restaurant POS this is implausible, and owner explicitly said *"We never show check in item in UI"* — which presumes name-based recognition is the operating definition. **Owner confirmation required before implementation** (see §11).

If owner provides the canonical `food_id` or `category_id` for the Check-In item from the POS configuration, we can add it as a stronger gate (Option B/C). Without that, A is the only feasible filter.

## 8. Proposed minimal fix

### Single change at the transformer (single source of truth)

`frontend/src/api/transformers/orderTransformer.ts` — inside `transformOrderDetails`:

```ts
// CR Fix: hide POS system item "Check In" from customer-facing UI.
// Backend already excludes it from order_amount / order_sub_total_amount,
// so this only filters the items list to remove the visual leak in
// OrderSuccess "Items Ordered" + ReviewOrder "Previously Ordered" + CartBar.
const isCheckInSystemItem = (api: ApiOrderDetailItem) => {
  const name = String(api.food_details?.name || '').trim().toLowerCase().replace(/[\s\-_]+/g, ' ');
  return name === 'check in';
};

const visibleDetails = (api.details || []).filter(d => !isCheckInSystemItem(d));
const items = visibleDetails.map(transformOrderItem);
const previousItems = visibleDetails.map(transformPreviousOrderItem);
```

That is the entire change. No other file touched. No payload, tax, calculation logic touched.

### Empty-state safety
After filtering, `items.length` and `previousItems.length` may be `0` (when an order contained ONLY Check In). The render guards in:
- `OrderSuccess.jsx:597` `{(isLoadingStatus || allItems.length > 0) && ...}`
- `PreviousOrderItems.jsx:90` `if (!items || items.length === 0) return null;`

…both handle this gracefully → the entire Items Ordered card is hidden, no misleading "Items Ordered (1)" with empty list. ✅

### itemTotal correction (auto)
`billSummary.itemTotal = previousItems.reduce(...)` → after filter, the ₹1 from Check In drops out. Aligns with `subtotal` and `grandTotal` from backend. No manual calc change, no risk to tax/SC/GST.
