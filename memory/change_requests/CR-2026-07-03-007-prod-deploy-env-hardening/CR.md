## 2. F-05 — Rebuild deployed bundle with correct backend URL

### Background
Live proof from DevTools during 2026-07-03 session:
```
Network tab on https://july-branch-deploy.preview.emergentagent.com/698 →
  GET https://cd356f08-6e6a-4e62-b619-b991a9d0fbcc.preview.emergentagent.com/api/config/698
```

That `cd356f08-…` host is **my dev container**. Whoever built the `july-branch-deploy` bundle used my session's `frontend/.env` which set `REACT_APP_BACKEND_URL` to my dev URL.

### Consequences today
- The deployed site cannot function if my dev container is stopped.
- CORS / origin telemetry attributed to the wrong environment.
- Data written from the deployed site's admin panel goes through my dev container's backend — currently that talks to prod Mongo, but the coupling is accidental.

### Proposed change
1. Identify the intended `REACT_APP_BACKEND_URL` for the `july-branch-deploy` environment (probably its own paired backend, or the prod FastAPI URL).
2. Update its build-time `.env`.
3. Rebuild the frontend bundle and redeploy.
4. Confirm DevTools Network on the deployed site shows API calls to the correct host.

### Files WILL change
- The **deploy pipeline's build-time env** (external to this repo, owned by whoever deploys). Not `/app/frontend/.env` in this repo.

### Files WILL NOT touch (in this repo)
- Any source code
- `/app/frontend/.env` here (it's for my local dev)

## 3. F-06 — Rotate all placeholders / dev secrets

### Background
Current state of `/app/frontend/.env` (dev):
```
REACT_APP_LOGIN_PHONE=<PLACEHOLDER_LOGIN_PHONE>
REACT_APP_LOGIN_PASSWORD=<PLACEHOLDER_LOGIN_PASSWORD>
REACT_APP_GOOGLE_MAPS_API_KEY=<PLACEHOLDER_DOMAIN_RESTRICTED_MAPS_KEY>
```

Current state of `/app/backend/.env` (dev):
```
JWT_SECRET=0436220...de98d4        ← freshly generated by me — DEV-ONLY
MONGO_URL=mongodb+srv://mygenie...  ← already prod Atlas creds, in git-tracked doc chain
```

### Proposed changes

| Secret | Action | Owner |
|---|---|---|
| `REACT_APP_LOGIN_PHONE` / `REACT_APP_LOGIN_PASSWORD` | Delete entirely once CR-2026-07-03-000 (login refactor) ships. Rotate the account in the CRM immediately regardless. | Security + Product |
| `REACT_APP_GOOGLE_MAPS_API_KEY` | Rotate in GCP + add HTTP referrer restrictions to prod domain(s). | Ops |
| `JWT_SECRET` | Rotate for prod. Never reuse dev value. Rotation forces all existing JWTs to expire — plan a maintenance window or use dual-signing during cutover. | Backend team |
| `MONGO_URL` | Rotate DB password (`mygenie_admin`). It has been in `DEPLOYMENT_HANDOVER.md` and this session logs. | DBA |
| `MYGENIE_API_URL` | Confirm this points at prod, not preprod, for a real production deploy. Current: `https://preprod.mygenie.online/api/v1` — likely wrong for prod. | Product |
| `REACT_APP_CRM_URL` | Same check — currently `https://crm.mygenie.online/api`. Verify prod endpoint. | Product |

### Files WILL change
- Production deployment's environment config (external to this repo).
- NOT `/app/backend/.env` or `/app/frontend/.env` here — those stay dev.

### Files WILL NOT touch
- Any source code
- Any `.env` in this repo (they're development-only)

### Risks
| # | Risk | Mitigation |
|---|---|---|
| R1 | JWT rotation invalidates existing admin sessions | Communicate window; dual-signing supports overlap if needed |
| R2 | Mongo password rotation causes brief downtime | Atlas supports rotate without downtime by adding new user first |
| R3 | Maps API key rotation breaks maps for tabs already open | Users refresh; brief interruption acceptable |
| R4 | POS/CRM URL flip changes data source unexpectedly | Confirm with product owner before flipping |
